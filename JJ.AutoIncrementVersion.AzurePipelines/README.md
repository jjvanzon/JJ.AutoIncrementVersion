BuildNum
========

__Azure Pipelines__ Task that:

- Finds `BuildNum.xml`
- Reads `BuildNum` element
- Stores it in the `$(BuildNum)` variable

It is intended for use with the [`JJ.AutoIncrementVersion`](https://www.nuget.org/packages/JJ.AutoIncrementVersion) nuget package, allowing auto-incremental version numbers for __.NET__ projects.

A build pipeline can benefit from this extension, by offering you the build number generated by `JJ.AutoIncrementVersion` as a `$(BuildNum)` variable. 

It can be used at your whim. For instance just to display it as part of the build/version/release numbers used in __Azure DevOps__.

But you could also pas it to a build task, e.g. with the `MSBuild Task` you can pass the `BuildNum` by filling this out under `MSBuild Arguments`:

```
/p:BuildNum=$(BuildNum)
```

This prevents the build from incrementing the `BuildNum` variable for each individual project in your solution. `MSBuild` has a way of deciding for us whether it'll use the initial build number or the new one, as multiple projects build one by one or even in parallel. That's why this Pipeline Task exists. It's a work around, but small and just enough to remove the problem.

It has no parameters, only the output `$(BuildNum)` variable, so you can just drop it in and use it.

If you need something more specific, I can recommend the [Set variable with value from XML](https://marketplace.visualstudio.com/items?itemName=YodLabs.VariableTasks) task by [Yod Labs](https://marketplace.visualstudio.com/publishers/YodLabs). Using these parameters accomplishes something similar:


```
Variable Name    : BuildNum
XPath expression : //BuildNum
XML file path    : BuildNum.xml
```

`Set variable with value from XML` might not recursively search for the `BuildNum.xml` but uses a literal path. `BuildNum` searches recursively and has no parameterization burden, working out of the box with the [`JJ.AutoIncrementVersion`](https://www.nuget.org/packages/JJ.AutoIncrementVersion) nuget package.
